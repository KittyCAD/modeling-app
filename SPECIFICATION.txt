Callback system for KCL execute

Goals
- Add execution-state callbacks to the execution context so consumers can observe progress while execution is still in flight.
- Start with a single callback type: `operation`.
- Fire the callback whenever `push_op` records an operation.
- Let efficient consumers build their own `OperationsByModule` maps incrementally instead of relying on `ExecOutcome.operations` for live updates.
- Keep existing `ExecOutcome.operations` behavior intact for compatibility.

Rust changes
- Add a callback holder to `ExecutorContext` in `rust/kcl-lib/src/execution/mod.rs`.
- The callback holder should support `on_operation(args)` where `args` contains `module_id`, `operation`, and `index`.
- Use obvious Rust types for those fields: current module id type, current operation type, and numeric index type.
- Ensure the callback holder is part of the context object, not an additional argument to `execute(...)`.
- Trigger `on_operation` from `ExecState::push_op` in `rust/kcl-lib/src/execution/state.rs`.
- Compute `index` from the owning module's operation list position at the time the operation is pushed.
- Preserve the existing stored operation list so `ExecOutcome.operations` is still produced normally.
- Keep callback invocation scoped to the module-local operation stream that `push_op` already appends to.

Wasm/context changes
- Extend the wasm `Context` implementation to store callback bindings alongside its other long-lived bridge state.
- Provide a way for TypeScript to register an object with `onOperation(args)` onto the context before execution.
- Do not add a new `execute(...)` parameter for callbacks; execution should read callbacks from the context.
- Translate Rust operation callback events across the wasm boundary into JS callback invocations.

TypeScript changes
- Add TypeScript types for the execution callback object and `onOperation(args: { moduleId, operation, index })` payload.
- Update the Rust/wasm wrapper layer so callbacks are registered on the shared context object.
- Update `RustContext` to accept or set this callback object before calling `execute(...)`.
- Keep existing `ExecState` / `ExecOutcome` usage working so current consumers are not forced to migrate immediately.

Behavior expectations
- `onOperation` fires once per `push_op` call.
- Callback ordering matches the per-module operation append order.
- `index` is stable within each module's operation stream.
- Imported-module operations report their own `module_id`.
- Consumers that want live operation state can reconstruct `OperationsByModule` from callbacks without waiting for final execution output.
- Existing consumers that only read `ExecOutcome.operations` after execution should continue to work unchanged.

Verification
- TypeScript should compile for touched files.
- Rust should compile for touched crates if feasible.
- Add focused tests for callback emission from `push_op` if there is a practical test seam.
- Verify callback registration on the wasm/TypeScript boundary if there is an existing test seam.

Refactor note
- Model execution callbacks as a trait-backed handler, similar to the wasm file-system adapter.
- Keep the JavaScript callbacks packed into one object while Rust forwards known callback methods through the trait.
- Preserve the public behavior of `onOperation(args)` on the TypeScript side.

Module state note
- Store the module id directly on `ModuleState` so callbacks and other module-local logic do not have to recover it from construction-only helper state.

Feature tree module ordering note
- Build the feature tree from the root module so imported module contents appear under their module instance instead of replacing the top-level tree when an imported file is open.
- Display module names for imported module nodes instead of the imported file path, and do not add a separate top-level `main.kcl` wrapper.
